# Project Agent Architect v2

Status: approved design; implementation requires separate explicit approval.

## 1. Job and non-goals

`project-agent-architect` bootstraps the smallest justified repository routing delta on top of a global best-value routing layer. It inspects evidence, checks repository-context readiness, proposes concise routing and exceptional local profiles, waits for approval, applies only approved files, and validates the result.

Non-goals:

- Generate a project team by default.
- Re-document or invent repository architecture.
- Duplicate engineering knowledge already supplied by skills.
- Mirror the full skill, MCP, or tool inventory into repository files.
- Require local scout, implementer, tester, researcher, or reviewer profiles.
- Silently migrate repositories when central policy changes.
- Provide platform-level automatic parent-model selection.

## 2. Smallest viable artifact set

Default repository output:

- A concise managed routing and repository-context block in `AGENTS.md`.
- Only `.codex/agents/*.toml` profiles justified by a materially distinct repository-local lane.

Not generated by default:

- A project-team skill.
- `evidence.md`.
- `architecture.md`.
- `agent-roster.md`.
- An intermediate team manifest.

`AGENTS.md` acts as an index. It points to authoritative architecture, security, workflow, and verification documents conditionally instead of duplicating them.

## 3. Model assignment

- Executable agent profiles pin exact model, reasoning, and sandbox values.
- Named model profiles such as `stable` or `beta` are not used.
- Central policy experiments use Git branches.
- Existing repositories change model assignments only through an explicit refresh delta.
- Missing models never fall back silently.

## 4. Delegation and escalation

The orchestrator owns classification, delegation, integration, escalation, and final judgment.

- Select the profile that best balances expected quality, cost, latency, access, and coordination overhead.
- Normal downward delegation does not require approval.
- Explicit user instructions override routing.
- The orchestrator may work directly when coordination would cost more than the work, but states the reason.
- Workers report evidence, confidence, blockers, and verification; they never promote themselves.
- A concrete Luna failure may escalate once to Terra automatically.
- Terra escalates to Sol only for conflicting evidence, architectural ambiguity, repeated failure, or security, concurrency, data, build, or deployment risk.
- Same-model retries occur only for environmental or transient failures.
- At most one Sol/high subagent runs automatically per user task. Further high-reasoning work requires approval.

Workflow selection and execution selection remain separate:

| Situation | Workflow |
|---|---|
| Clear single-session change | `implement` |
| Clear plan needing multiple tickets | `to-tickets` |
| Idea needing clarification | `grill-with-docs` |
| Grilled multi-session build | `to-spec` then `to-tickets` |
| Huge effort whose route remains unclear | `wayfinder` |

`implement` drives `tdd` at agreed seams and closes with `code-review`. Use `tdd` directly only for a concrete test-first change that does not need the spec or ticket flow.

## 5. Minimum-team framework

Repository size never justifies a profile by itself.

A repository-local profile requires all of:

1. A task class recognizable before work starts.
2. A distinct model, reasoning, sandbox, tool, or verification boundary.
3. A concise and stable repository-specific instruction delta.
4. Expected quality or context value greater than delegation overhead.

It also requires evidence from repository history, instructions, plans, or explicit user confirmation that the lane recurs or protects a repeatedly important independent boundary.

| Level | Result |
|---|---|
| 0 | Global profiles plus repository routing block; no local profiles |
| 1 | One proven local profile |
| 2 | Separate local read/write lanes when both are justified |
| 3 | Specialists only after demonstrated recurring need |

## 6. Source-of-truth map

| Concern | Owner |
|---|---|
| Best-value selection and escalation | Central `routing-policy.md` |
| Automatically loaded global routing summary | Managed block in global `AGENTS.md` |
| Repository context, pointers, and exceptions | Repository `AGENTS.md` |
| Architecture and security decisions | Repository architecture/security docs and ADRs |
| Model, reasoning, sandbox, and agent instructions | Agent TOML |
| Engineering expertise and methods | Composable skills |
| Runtime classification and escalation | Orchestrator |
| Structural correctness | Shared validator |
| Initial repository setup | `project-agent-architect` |
| Later capability/model changes | `refresh-project-agent-routing` |

The global `AGENTS.md` block is a concise runtime projection of central policy. The validator detects drift between them.

## 7. Validator scope

The validator checks deterministic facts only.

`FAIL`:

- Invalid TOML or skill frontmatter.
- Duplicate agent names.
- Routing references a nonexistent profile.
- A confirmed unavailable model is assigned.
- A missing skill/tool is claimed as available.
- Sandbox contradicts an explicit access requirement.
- Refresh touches files outside the approved list or managed section.

`WARN`:

- Model availability cannot be verified.
- A profile has no routing trigger.
- A repository-context pointer is missing.
- Instructions repeat substantial skill or repository documentation.
- A profile appears broader than its stated responsibility.
- Non-critical repository context remains unresolved.

`PASS` means configuration is structurally valid and referenced surfaces are observed. It does not prove routing quality, repository architecture quality, or economic value.

Failures are fixed before application. Warnings may be fixed, explicitly accepted, or deferred. The skill revalidates in the same session; users do not normally rerun it manually.

## 8. Required failure scenarios

| Scenario | Required handling |
|---|---|
| An upstream update removes or renames `to-prd` and `to-issues` | Refresh fails stale references, proposes `to-spec` and `to-tickets`, and preserves unrelated routing and models |
| A new repository has a plan but little documentation or history | Preflight uses plans, manifests, scripts, tests, and user confirmation; proposes only evidence-backed context and blocks unsafe unknown routes |
| Luna misunderstands cross-component work | Orchestrator identifies a concrete omission, escalates to Terra, and uses Sol only for remaining ambiguity, repeated failure, or elevated risk |

## 9. v2 change proposal

### Must change

- Split bootstrap and refresh into two user-facing skills with shared policy and validation.
- Replace team generation with routing bootstrap.
- Introduce global profiles plus repository-local deltas.
- Add repository-context preflight.
- Use exact model assignments and remove named model profiles.
- Replace the long draft with a concise approval surface.
- Add managed ownership, minimal patching, and deterministic validation.
- Permit `no local profiles needed` as a successful result.
- Make the orchestrator own bounded escalation.
- Add an explicit global-routing migration before repository experiments.

### Must cut

- Mandatory project-team skill and project-profile documents.
- Mandatory role roster.
- Full capability-inventory dumps.
- Repeated skill methodology inside profile prompts.
- Blanket delegation of all non-trivial work.
- Persona-only specialization.
- Bootstrap/refresh mode inference.
- Whole-file regeneration.

### Should change

- Add compact managed-section markers.
- Require each profile to state trigger, access, output, and composed skills.
- Offer audit-only validation through the shared script, not another user-facing skill.
- Record routing decisions during initial experiments.

### Deliberately deferred

- General `repo-context-bootstrap` skill.
- Intermediate manifests.
- Automatic repository migrations.
- Cost dashboards and telemetry infrastructure.
- Deterministic economic scoring.
- Large specialist catalogs.

## Approval surface

Default bootstrap and refresh output contains only:

1. PASS/WARN/FAIL status and recommendation.
2. Routing table: agent, trigger, model, reasoning, access, skills.
3. Attention-required model, role, access, capability, and context changes.
4. File delta.
5. Approval checklist.

Detailed evidence and rejected alternatives remain available on demand. Refresh shows only changed rows plus `Everything else preserved`.

Success:

- Concise primary approval table plus attention items.
- Defensible best-value routing and no unsupported surfaces.
- Every surviving local profile passes the admission test.
- No silent fallback, overwrite, or unrelated change.
- Apple specialists appear only for Apple-specific value.
- At least seven of nine simulated routes require no user correction.

Hard failure:

- Unsafe write access, missed boundary, unsupported capability, overwritten manual edit, silent model substitution, unrelated refresh change, or automatic repository migration.

Design failure:

- More than two of nine routes require correction.
- Cheap routing rework erases savings.
- Approval drafts remain difficult to review.
- Global profiles repeatedly require duplicated repository prompts.
- Most repositories still receive large local teams.
