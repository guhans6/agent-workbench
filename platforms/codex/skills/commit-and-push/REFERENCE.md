# Commit And Push Reference

## Purpose

This skill handles publish hygiene after implementation work.
It is not a generic PR bot, release manager, or issue triage system.

## Scope Inference

Default source order:

1. current conversation request
2. repo-local workflow or issue-tracker docs
3. git diff truth
4. tracker state when needed

Current conversation is enough to infer task scope, not enough by itself to justify issue refs.
For issue linkage, require explicit issue IDs or repo/tracker evidence.
Do not pretend the diff is cleaner than it is.
If the worktree is mixed and clean isolation is not obvious, stop and ask.
If issue mapping stays ambiguous after inspection, omit issue refs and report that the mapping was not reliable, or stop and ask when issue-aware publishing was explicitly requested.

## Commit Grouping Standard

Preferred grouping:

- one commit when the change is one coherent unit
- two or a few commits when there is a clean split such as:
  - refactor + behavior fix
  - migration + follow-up cleanup
  - test infrastructure + covered behavior changes
  - generated files paired with their source change when separation would make history worse

Avoid:

- one giant review-unit dump when the history would hide distinct intent
- tiny mechanical micro-commits that bloat history without adding meaning
- grouping unrelated issue work into one commit without explanation

If multiple issue-driven slices share unavoidable infrastructure code, either:

- create separate commits per issue plus one shared foundation commit with `Refs:` for each relevant issue, or
- explain in the commit body why a shared commit is cleaner than artificial separation

## Commit Message Standard

Use Conventional Commits:

```text
<type>[optional scope]: <summary>

[optional body]

[optional footer(s)]
```

Types:

- `feat`
- `fix`
- `refactor`
- `perf`
- `test`
- `docs`
- `build`
- `ci`
- `chore`
- `revert`

Subject rules:

- imperative mood
- concise
- no trailing period
- no AI attribution
- no issue numbers by default

Body rules:

- include only when needed for rationale, behavior change, migration, rollback risk, or non-obvious scope
- describe why over what
- keep it readable, not padded

Footer rules:

- use the repo's issue-reference convention for linked issues by default, for example `Refs: #123`
- allow multiple issue refs when genuinely needed
- use trailer-style metadata for structured fields

Recommended default:

- no issue refs in subject
- no closing keywords in commits by default
- issue closure lives in the PR body

## PR Boundary

After push, stop and ask before creating or updating a PR.
Pushing commits to a branch that already has a PR is allowed by default.
Treat "PR update" here as metadata, title, body, labels, comments, or other explicit PR edits.

The post-push summary should include:

- branch name
- commit list
- linked issues
- whether the branch already has an open PR
- whether any issue should be closed on merge
- recommended PR title
- short PR summary
- observed validation or `unverified`

If the user approves PR creation or update:

- prefer a concise senior-style PR title
- explain what changed, why, risk, and validation
- use issue-closing keywords in the PR body when merge should close the issue

## Hard Prohibitions

Never:

- add "Generated by", "Authored by agent", or similar text
- commit unrelated changes silently
- use `git add -A` on a mixed worktree without clear scope
- close issues from commit messages by default
- invent validation that did not run
- recover from push rejection with pull, rebase, merge, or force-push unless the next step is clearly safe and in scope
- create or update a PR without explicit approval after push

## Push Safety

Default-branch or protected-branch pushes require explicit user approval after the target branch is made clear.
Do not treat missing branch protection or permissive repo settings as approval.

For no-upstream branches:

- if there is one clear writable remote and normal branch publishing is expected, creating the upstream is allowed
- if multiple remotes or unclear targets exist, stop and ask
- do not guess between fork and upstream remotes

## Examples

Small scoped fix:

```text
fix(editor): preserve selection after rename

Refs: #142
```

Larger change with rationale:

```text
feat(sync): retry transient upload failures

Retry idempotent upload requests with bounded backoff and keep
permanent failures visible to the user.

Refs: #188
```

PR body after approval:

```md
## Summary
- retry transient upload failures
- preserve permanent error state
- add coverage for retry classification

## Validation
- ran unit tests for sync client
- verified retry path manually

Closes #188
```
